# 交付物一：分组合并联表（分组驱动结构）

> **结论先行**：等保三级安全计算环境 13 个控制点并非平均适用于所有服务器类型。按系统角色分组管理，可减少 40%-60% 的重复检查项，将分散的 127 个要求项（三级全部层面）压缩为每组 30-50 个针对项。

---

## 1. 分组策略

### 分组依据

按操作系统类型 + 承载服务角色，将主机分为 5 组：

| 分组 | 典型系统 | 覆盖范围 | 适用控制点数量 |
|:---|:---|:---|:---|
| **G1 — Linux 服务器** | CentOS 7/8, Rocky Linux, Ubuntu Server, EulerOS | Web服务器、应用服务器、数据库服务器（OS层） | 10 个控制点 |
| **G2 — Windows 服务器** | Windows Server 2016/2019/2022 | AD域控、文件服务器、业务管理系统 | 11 个控制点 |
| **G3 — 数据库** | Oracle 19c, MySQL 8.0, SQL Server 2019, PostgreSQL, Redis | 承载业务数据的数据库实例 | 8 个控制点 |
| **G4 — 中间件/应用** | Tomcat, Nginx, Apache, WebLogic, JBoss, Node.js | Web应用服务器、API网关 | 6 个控制点 |
| **G5 — 网络设备** | 交换机(华为CE/S5700)、路由器、防火墙 | 网络基础架构 | 另见网络安全章节 |

### 2.2 分组内详细控制点清单

#### G1 — Linux 服务器（10 控制点, 42 要求项）

```
身份鉴别(S3)  — 7项：唯一性、标识鉴别、双因素、复杂度、登录失败、超时退出、鉴别警示
访问控制(S3)  — 9项：自主访问控制、强制访问控制(SELinux)、权限分离、最小权限、默认用户
安全审计(G3)  — 5项：审计覆盖、事件类型、记录要素、审计分析、审计保护
入侵防范(G3)  — 5项：最小安装、关闭端口/服务、漏洞管理、入侵检测/文件完整性、内核参数
恶意代码防范    — 2项：ClamAV/防病毒、特征库更新（服务端不需要，Web/文件服务器需要）
资源控制(A3)  — 4项：登录超时、会话限制、资源限制(ulimit)、CPU/内存/磁盘阈值告警
数据完整性(S3) — 3项：传输完整性(SSH/HTTPS)、存储完整性(AIDE/Tripwire)、校验机制
数据保密性(S3) — 3项：传输加密(SSH/HTTPS/LUKS)、存储加密(LUKS)、加密算法配置
剩余信息保护    — 2项：内存清除(shred)、磁盘清理
集中管控(G3)   — 6项：统一身份管理、审计日志集中、安全事件汇集、策略一致性、补丁管理、运维审计
```

#### G2 — Windows 服务器（11 控制点, 44 要求项）

```
身份鉴别(S3)  — 7项：密码策略(secpol.msc)、账户锁定策略、智能卡/Kerberos、密码历史、交互式登录
访问控制(S3)  — 9项：NTFS权限、UAC、用户权限分配、安全选项、服务权限、注册表权限
安全审计(G3)  — 5项：审核策略(secpol.msc)、高级审核配置(4624/4625)、事件日志大小/保留、日志转发(WEF)
入侵防范(G3)  — 5项：(同Linux + Windows Defender)
恶意代码防范    — 2项：Windows Defender/第三方防病毒、实时保护、定期扫描
资源控制(A3)  — 4项：远程桌面超时(Group Policy)、终端服务限制、进程资源限制
数据完整性(S3) — 3项：传输完整性(RDP/File Share)、存储完整性(DFS Replication校验)
数据保密性(S3) — 3项：BitLocker(OS/数据盘)、EFS、传输加密(RDP 128位)
剩余信息保护    — 2项：cipher /w、回收站策略、页面文件清除
个人信息保护    — 2项：含PII的Windows系统(域控含用户信息)
集中管控(G3)   — 6项：AD统一管理、GPO统一配置、WSUS补丁、SCOM/System Center
```

#### G3 — 数据库（8 控制点, 30 要求项）

```
身份鉴别(S3)  — 7项：账户唯一性、密码策略(validate_password)、双因素(可选)、连接加密、登录失败
访问控制(S3)  — 9项：权限最小化(DBA/只读/应用账户)、角色分离、存储过程权限、视图权限
安全审计(G3)  — 5项：审计日志(audit_log/binlog)、DDL/DML审计、日志保留、审计表保护
资源控制(A3)  — 4项：max_connections、连接超时(wait_timeout)、查询超时、磁盘空间告警
数据完整性(S3) — 3项：数据文件校验(checksum table)、传输完整性(TLS)、主从复制校验
数据保密性(S3) — 3项：TLS加密连接(require_secure_transport)、TDE透明加密、字段级加密
数据备份恢复    — 3项：RPO/RTO定义、全量+增量备份策略、备份验证(定期恢复演练)
个人信息保护    — 2项：含PII的表字段加密/脱敏、动态脱敏(如MySQL Enterprise Audit)
```

#### G4 — 中间件/应用（6 控制点, 20 要求项）

```
身份鉴别(S3)  — 3项：管理控制台认证、默认账户删除(changed)、会话超时
访问控制(S3)  — 3项：目录权限控制、最小功能(禁用不安全HTTP方法/目录列表)、应用级RBAC
安全审计(G3)  — 5项：访问日志(access_log)、错误日志(error_log)、审计格式(combined)、日志轮转
资源控制(A3)  — 4项：连接池限制/worker限制、超时配置、请求体大小限制、并发限制
数据完整性(S3) — 3项：传输完整性(HTTPS/HSTS/证书验证)、内容校验
软件容错(A3)  — 2项：会话保持/集群、优雅降级/健康检查
```

#### G5 — 网络设备（控制点见网络安全章节，此处仅列主机相关的集中管控）

```
集中管控(G3)   — 6项：SSH/Telnet策略、TACACS+/RADIUS认证、NTP同步、syslog集中、SNMPv3
```

## 3. 分组并联的优势

### 3.1 避免重复检查

传统做法：对所有 127 个要求项逐一检查每台服务器，产生大量 N/A（不适用）标记。

分组并联后：
- Linux 服务器只检查 42 项（跳过数据备份恢复、恶意代码防范、个人信息保护等）
- 数据库只检查 30 项（跳过入侵防范、恶意代码防范、剩余信息保护、软件容错等）
- 中间件只检查 20 项（聚焦 Web 安全相关）

### 3.2 提高整改效率

| 阶段 | 传统逐项法 | 分组并联法 | 优化幅度 |
|:---|:---|:---|:---:|
| 检查工作量 | 127项/台 × N台 | (42+30+20)/3 × N台 | ~40% |
| 整改指令产出 | 逐系统下发 | 按组一次下发 | ~60% |
| 报告篇幅 | 大量 N/A 冗余 | 按组呈现清晰 | ~50% |

### 3.3 与 ISEAA 测评要求的关系

分组并联并不改变 ISEAA 测评的单元结构，而是为**整改实施阶段**提供工程化分组策略。测评方仍按传统层面+控制点结构出具报告。本表用于：

1. **售前/方案设计**：快速评估每个系统组的整改工作量
2. **实施部署**：按组批量下发脚本和配置
3. **项目管理**：分组长周期跟踪进度（如G1完成80%，G3完成60%）

## 4. 典型项目应用示例

假设某政务云平台共 30 台主机：

| 分组 | 数量 | 系统 | 检查项数 | 预估工时(天) |
|:---|:---:|:---|:---:|:---:|
| G1 Linux | 18 | 应用服务器 + Web服务器 | 42 | 4 |
| G2 Windows | 5 | AD域控 + 文件服务器 | 44 | 3 |
| G3 数据库 | 4 | Oracle + MySQL | 30 | 3 |
| G4 中间件 | 3 | Tomcat + Nginx | 20 | 1 |
| **合计** | **30** | — | — | **~11** |

> 注：工时含检查+整改+验证，双人并行实际约6-7工作日。
